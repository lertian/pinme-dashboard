# 需求文档

## 简介

本功能旨在为 PinMe 项目添加用户认证和项目管理功能。目前 PinMe 支持文件上传到 IPFS，但用户无法追踪和管理已上传的项目。通过添加用户系统和项目管理界面，用户可以：

1. 注册账号并登录
2. 查看所有已上传的项目
3. 为项目重新上传新版本
4. 通过卡片式界面直观地管理所有项目

该功能将基于现有的 Supabase 示例项目进行扩展，保持现有的上传流程和工作方式不变。

---

## 需求

### 需求 1：用户认证系统

**用户故事：** 作为一个 PinMe 用户，我希望能够通过用户名和密码注册账号并登录，以便我能够安全地管理我的上传项目。

#### 验收标准

1. **WHEN** 用户访问应用首页 **THEN** 系统应显示登录/注册界面
2. **WHEN** 用户在注册表单输入用户名、密码和确认密码 **THEN** 系统应验证：
   - 用户名唯一性（未被使用）
   - 用户名长度（3-20个字符）
   - 密码长度（至少6个字符）
   - 密码和确认密码一致
3. **WHEN** 用户提交注册表单且验证通过 **THEN** 系统应创建新账户并自动登录
4. **WHEN** 用户输入已注册的用户名和密码点击登录 **THEN** 系统应验证凭据并允许访问
5. **WHEN** 用户已登录 **THEN** 系统应在所有页面显示用户名和登出按钮
6. **WHEN** 用户点击登出 **THEN** 系统应清除会话并返回登录页面
7. **WHEN** 普通用户登录 **THEN** 系统应显示项目管理界面
8. **WHEN** 管理员用户登录 **THEN** 系统应显示管理员控制面板入口

### 需求 1.1：用户密码管理

**用户故事：** 作为一个已登录的用户，我希望能够修改自己的密码，以便保护账户安全。

#### 验收标准

1. **WHEN** 用户在个人设置页面 **THEN** 系统应显示"修改密码"选项
2. **WHEN** 用户点击"修改密码" **THEN** 系统应显示密码修改表单，包含：
   - 当前密码输入框
   - 新密码输入框
   - 确认新密码输入框
3. **WHEN** 用户提交密码修改表单 **THEN** 系统应验证：
   - 当前密码正确
   - 新密码长度（至少6个字符）
   - 新密码和确认密码一致
   - 新密码与当前密码不同
4. **WHEN** 验证通过 **THEN** 系统应更新密码并显示成功提示
5. **WHEN** 验证失败 **THEN** 系统应显示具体错误信息

### 需求 1.2：管理员用户管理

**用户故事：** 作为管理员，我希望能够查看所有注册用户并为他们重置密码，以便帮助用户解决账户问题。

#### 验收标准

1. **WHEN** 管理员登录 **THEN** 系统应在导航栏显示"用户管理"入口
2. **WHEN** 管理员访问用户管理页面 **THEN** 系统应显示所有注册用户列表，包含：
   - 用户名
   - 注册时间
   - 最后登录时间
   - 项目数量
   - 操作按钮（重置密码、查看详情）
3. **WHEN** 管理员点击"重置密码"按钮 **THEN** 系统应显示密码重置对话框
4. **WHEN** 管理员在对话框中输入新密码并确认 **THEN** 系统应：
   - 验证新密码长度（至少6个字符）
   - 更新用户密码
   - 显示成功提示
5. **WHEN** 管理员点击"查看详情" **THEN** 系统应显示该用户的详细信息和所有项目
6. **WHEN** 非管理员用户尝试访问用户管理页面 **THEN** 系统应拒绝访问并显示权限不足提示

### 需求 2：项目列表管理

**用户故事：** 作为一个已登录的用户,我希望能够查看我所有已上传的项目列表，以便我能够追踪和管理我的所有部署。

#### 验收标准

1. **WHEN** 用户登录成功 **THEN** 系统应显示该用户所有已上传项目的卡片式列表
2. **WHEN** 项目列表为空 **THEN** 系统应显示空状态提示和"上传新项目"按钮
3. **WHEN** 项目列表不为空 **THEN** 每个项目卡片应显示：
   - 项目名称
   - 上传时间
   - IPFS CID
   - 预览链接
   - 固定域名（如果有）
   - 版本号
4. **WHEN** 用户点击项目卡片 **THEN** 系统应显示项目详情和操作选项
5. **WHEN** 用户滚动到列表底部 **AND** 还有更多项目 **THEN** 系统应自动加载更多项目

### 需求 3：新项目上传

**用户故事：** 作为一个已登录的用户，我希望能够上传新项目到 IPFS，以便部署我的静态网站。

#### 验收标准

1. **WHEN** 用户点击"上传新项目"按钮 **THEN** 系统应显示上传对话框
2. **WHEN** 用户在上传对话框中 **THEN** 系统应允许用户：
   - 输入项目名称
   - 选择要上传的文件夹
   - （可选）指定固定域名
3. **WHEN** 用户选择文件夹 **THEN** 系统应验证文件夹包含 index.html
4. **WHEN** 用户点击确认上传 **THEN** 系统应：
   - 显示上传进度
   - 将文件上传到 IPFS
   - 保存项目信息到数据库（关联到当前用户）
   - 返回 IPFS CID 和预览链接
5. **WHEN** 上传成功 **THEN** 系统应在项目列表中显示新项目
6. **WHEN** 上传失败 **THEN** 系统应显示错误信息并允许重试

### 需求 4：项目版本更新

**用户故事：** 作为一个已登录的用户，我希望能够为已有项目上传新版本，以便更新我的网站内容而不丢失历史记录。

#### 验收标准

1. **WHEN** 用户在项目卡片上点击"更新版本"按钮 **THEN** 系统应显示版本更新对话框
2. **WHEN** 用户在版本更新对话框中 **THEN** 系统应：
   - 显示当前项目信息
   - 允许选择新的文件夹
   - 允许输入版本说明
3. **WHEN** 用户选择文件夹并确认 **THEN** 系统应：
   - 上传新文件到 IPFS
   - 保存新版本记录（包含新的 CID、时间戳、版本说明）
   - 更新项目的当前版本指向新版本
   - 保留旧版本的历史记录
4. **WHEN** 版本更新成功 **THEN** 系统应：
   - 更新项目卡片显示新版本信息
   - 增加版本号
5. **WHEN** 用户点击项目详情中的"版本历史" **THEN** 系统应显示所有历史版本列表

### 需求 5：项目详情和操作

**用户故事：** 作为一个已登录的用户，我希望能够查看项目的详细信息并执行各种操作，以便更好地管理我的项目。

#### 验收标准

1. **WHEN** 用户点击项目卡片 **THEN** 系统应显示项目详情页面，包含：
   - 项目基本信息（名称、创建时间、最后更新时间）
   - 当前版本的 IPFS CID
   - 预览链接（可点击）
   - 固定域名（如果有）
   - 版本历史列表
2. **WHEN** 用户在项目详情页 **THEN** 系统应提供以下操作按钮：
   - "更新版本"
   - "复制 CID"
   - "复制预览链接"
   - "在新标签页打开预览"
   - "删除项目"
3. **WHEN** 用户点击"复制 CID" **THEN** 系统应复制 CID 到剪贴板并显示成功提示
4. **WHEN** 用户点击"复制预览链接" **THEN** 系统应复制预览链接到剪贴板并显示成功提示
5. **WHEN** 用户点击"在新标签页打开预览" **THEN** 系统应在新标签页打开预览链接
6. **WHEN** 用户点击"删除项目" **THEN** 系统应：
   - 显示确认对话框
   - 在用户确认后从数据库删除项目记录
   - 从项目列表中移除该项目
7. **WHEN** 用户在版本历史中点击某个历史版本 **THEN** 系统应显示该版本的详细信息和预览链接

### 需求 6：卡片式界面设计

**用户故事：** 作为一个已登录的用户，我希望通过美观直观的卡片式界面管理项目，以便快速找到和操作我的项目。

#### 验收标准

1. **WHEN** 用户查看项目列表 **THEN** 系统应以网格布局显示项目卡片
2. **WHEN** 屏幕宽度较大 **THEN** 系统应显示 3 列卡片
3. **WHEN** 屏幕宽度中等 **THEN** 系统应显示 2 列卡片
4. **WHEN** 屏幕宽度较小（移动设备） **THEN** 系统应显示 1 列卡片
5. **WHEN** 用户鼠标悬停在项目卡片上 **THEN** 系统应显示悬停效果（阴影、轻微放大等）
6. **WHEN** 项目卡片显示 **THEN** 应包含：
   - 项目图标或缩略图
   - 项目名称（大字体）
   - 最后更新时间（相对时间，如"2天前"）
   - 版本号徽章
   - 快速操作按钮（更新、预览、更多）
7. **WHEN** 用户点击卡片上的快速操作按钮 **THEN** 系统应执行相应操作而不跳转到详情页

### 需求 7：数据持久化和同步

**用户故事：** 作为一个用户，我希望我的项目数据能够安全存储并在不同设备间同步，以便我可以在任何地方访问我的项目。

#### 验收标准

1. **WHEN** 用户上传新项目 **THEN** 系统应将项目信息保存到 Supabase 数据库
2. **WHEN** 用户更新项目版本 **THEN** 系统应保存新版本记录到数据库
3. **WHEN** 用户在不同设备登录同一账号 **THEN** 系统应显示相同的项目列表
4. **WHEN** 数据库操作失败 **THEN** 系统应显示友好的错误信息并提供重试选项
5. **WHEN** 用户离线时 **THEN** 系统应显示离线提示并缓存操作，待网络恢复后同步
6. **WHEN** 用户删除项目 **THEN** 系统应软删除记录（标记为已删除）而不是物理删除

### 需求 8：搜索和筛选

**用户故事：** 作为一个拥有多个项目的用户，我希望能够搜索和筛选项目，以便快速找到特定项目。

#### 验收标准

1. **WHEN** 用户在项目列表页 **THEN** 系统应显示搜索框
2. **WHEN** 用户在搜索框输入文本 **THEN** 系统应实时过滤项目列表（按项目名称匹配）
3. **WHEN** 用户清空搜索框 **THEN** 系统应显示所有项目
4. **WHEN** 用户点击排序选项 **THEN** 系统应允许按以下方式排序：
   - 最近更新（默认）
   - 创建时间
   - 项目名称（字母顺序）
5. **WHEN** 搜索或筛选后无结果 **THEN** 系统应显示"未找到项目"提示

## 边界情况和约束

### 边界情况

1. **匿名上传的历史记录**：现有系统支持匿名上传，需要考虑如何处理用户注册前的匿名上传记录
2. **大量项目**：用户可能有数百个项目，需要分页或虚拟滚动
3. **并发上传**：用户可能同时上传多个项目
4. **网络中断**：上传过程中网络中断需要支持断点续传或重试
5. **域名冲突**：多个用户可能尝试绑定相同的域名

### 技术约束

1. 必须使用 Supabase 作为后端（认证和数据库）
2. 必须保持现有的 PinMe CLI 上传流程不变
3. 必须使用 React + TypeScript + Vite
4. 必须支持响应式设计（移动端和桌面端）
5. 必须保持与现有 IPFS 网关的兼容性

### 性能要求

1. 项目列表加载时间应小于 2 秒
2. 搜索响应时间应小于 500 毫秒
3. 上传进度应实时更新（至少每秒一次）

### 安全要求

1. 所有 API 请求必须经过身份验证
2. 用户只能访问自己的项目
3. 密码必须使用 bcrypt 或类似算法加密
4. 必须防止 SQL 注入和 XSS 攻击
